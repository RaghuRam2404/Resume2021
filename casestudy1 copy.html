Zoho CRM API V2
During the time period before 2015, Zoho CRM offered only some restricted list of APIs in a non-standard way (i.e. no standard API URI structure, different type of responses for same resource across different APIs). So by the mid of 2015, we embarked on a journey to entirely revamp in terms of
API URL
 
Request/Response structure
 
OAuth2.0 Authorization
 
Multi DC Handling
 
Exposing more functionalities via API
 
More API Limits, but with proper rate limiting

Technical Improvements
1. More API Credits

From having very low API limits in the older API version, we gave more API limits for customer's best usage of our APIs. We introduced API Credits so that resource intensive API calls will use more API Credits and less intensive requests will use lesser API credits. We adopted 2 methodologies for Rate Limiting and Throttling.

Rolling Window Throttling for 24 hours
 

We used to throttle the API calls for any day from PDT 00:00:00 to 23:59:59 . We changed it to rolling window of 24 hours. At any instance of time, we allowed only to fire API calls within the allowed API credits based on the last 24 hours usage. We achieved it using Redis (due to it's single thread nature and faster access).

Concurrency Throttling
 

It defines the number of API calls which can be active at any given time instance. This is done to avoid the DDOS attack of API calls in our servers. If any customer fires a resource intensive API call in parallel, it'll occupy many threads and causes resource deficit to other API calls. We'll arrest it by concurrency.

2. Faster API Responses

Since we planned to give more API Credits to the user, we must optimize the response time and apply concurrency over that. We have applied necessary caching for queries and removed unwanted functionalities provided in the older API version.

3. New APIs

We have published new APIs like Notifications API (pub/sub), Bulk API (to push & pull data in bulk). They acted as selling points for the developers to use the new APIs.

4. Single Code Logic

We used to have 2 different set of code logics, one for real business usecase and another for APIs. As time moves on, we can't keep track of new business logics and existing business logic change. So we changed the code to use the native business logic for better consistency.

Challenges & Solutions
1. Communication to Customers

As we have released a major version and we want our customers to migrate faster, we have to communicate across different channels to get the attention of third party app developers, CRM's partners and CRM's customers. The communication channel includes email, phone, in-product banner, public forum posts and followed up with multiple reminders in those channels.

2. API Migration

The changes needed to do this migration is very big and it'll be time consuming. We have to present with proper documentation and help, so that it'll be easy to refer and do the code migration in all different types of usecases like normal API calls, webhooks etc.
